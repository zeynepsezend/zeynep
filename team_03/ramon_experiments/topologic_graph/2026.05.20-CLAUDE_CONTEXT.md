## 1. Resumen del proyecto principal (MASTER_CLAUDE.md)

El proyecto **Team 03 - Accessibility Agent** es un agente de IA que evalua plantas arquitectonicas (residenciales e industriales) en terminos de accesibilidad espacial. Su arquitectura:

- **Input:** Un JSON de layout con 7 capas (outline, rooms, doors, windows, furniture, mep, structure) + un prompt en lenguaje natural.
- **Preprocesamiento:** Dos agentes LLM (Profile Agent + Space Type Agent) enriquecen el contexto con restricciones del usuario (wheelchair, elderly, forklift, etc.) y prioridades del espacio.
- **Loop principal (LangGraph):** El nodo `reason` decide si colocar objetos, llamar herramientas MCP (Grasshopper/Rhino), o finalizar.
- **Pipeline de analisis (6 herramientas):**
  1. **Collision** (peso 0.30) - Grid BFS, clearance, turning radius
  2. **Path Analysis** (peso 0.25) - BFS entre rooms + A* dentro de rooms
  3. **Visibility** (peso 0.20) - Linea de vision con Shapely
  4. **Reachability** (peso 0.15) - Envolvente ergonomica de alcance
  5. **Orientation** (peso 0.10) - Direccion de orientacion de objetos
  6. **Scoring** - Score ponderado 0-100 con grado A-F
- **Output:** Layout final aprobado por el usuario, con score y explicacion.

El agente opera en un ciclo iterativo: analiza, propone cambios, el usuario aprueba o pide ajustes.

## 2. Resumen del script de grafos (CLAUDE_GRAPH.md)

El script `layout_graph_generator.py` de mi companero toma el **mismo formato JSON de layout** y lo convierte en un **grafo de red (NetworkX)** para analisis topologico:

### Input
- JSON de layout arquitectonico (mismas 7 capas del proyecto principal)

### Proceso (4 fases)
1. **Deteccion de elementos:** Parsea todas las categorias, extrae geometria/centro/bounding box, crea un nodo por elemento.
2. **Descubrimiento de relaciones (5 tipos de aristas):**
   - `references` - Atributos que referencian IDs de otros elementos
   - `spatial_near` - Proximidad euclidiana < 15 unidades (normalizada 0-1)
   - `contained_in` - Elementos con roomId
   - `system_group` - Elementos del mismo sistema MEP
   - `door_connection` - Puertas que conectan habitaciones
3. **Space Syntax Metrics:** Degree, betweenness, closeness, eigenvector centrality por nodo.
4. **Exportacion:** PNG del grafo + 3 CSVs (nodes, edges, metrics_summary)

### Output
- Visualizacion estatica (PNG 26x20in, 300 DPI)
- CSVs con metricas de centralidad y relaciones

## 3. Puntos de convergencia y diferencias

| Aspecto | Accessibility Agent | Graph Generator |
|---------|-------------------|-----------------|
| **Input** | Mismo JSON de layout | Mismo JSON de layout |
| **Relaciones que detecta** | Doors-rooms (BFS), collision grid, visibility lines, paths | 5 tipos: references, spatial_near, contained_in, system_group, door_connection |
| **Metricas** | Score de accesibilidad (0-100) por herramienta | Centralidades de grafos (degree, betweenness, closeness, eigenvector) |
| **Proposito** | Evaluar y mejorar accesibilidad iterativamente | Analizar y visualizar la topologia relacional del edificio |
| **Interactividad** | Si (loop con usuario, checkpoint, viewport) | No (script one-shot, exporta archivos) |
| **Dependencias externas** | Grasshopper/Rhino via MCP, LLM | Solo Python (networkx, matplotlib) |

**Overlap critico:** Ambos sistemas descubren relaciones entre elementos del layout, pero con enfoques distintos. El agent se enfoca en *accesibilidad funcional* (puede una persona pasar, alcanzar, ver). El graph generator se enfoca en *topologia relacional* (que esta conectado a que, que es central en la red).

## 4. Posible integracion del grafo en el workflow del agente

### 4.1 Como input enriquecido para el agente

El grafo topologico podria alimentar al agente **antes** del loop principal, como un tercer agente de preprocesamiento (junto a Profile Agent y Space Type Agent):

- **Betweenness centrality alta** en una habitacion = es un "puente" critico en la circulacion. El agente deberia priorizar su analisis de paths y collision.
- **Degree centrality alta** = muchas conexiones, nodo congestionado. Mas relevante para collision analysis.
- **Closeness centrality baja** = espacio aislado, posible problema de evacuacion/egreso.
- **Eigenvector centrality** = influencia en la red, util para identificar los espacios mas "importantes" del layout.

### 4.2 Como capa de validacion post-analisis

Despues de que el agente coloque objetos y corra el pipeline de analisis, el grafo podria recalcularse para verificar:
- Si los objetos colocados alteraron las relaciones espaciales (nuevas aristas `spatial_near`, eliminacion de paths).
- Comparar la topologia "antes vs despues" del layout modificado.

### 4.3 Como herramienta de decision para el reason node

Las metricas de space syntax podrian inyectarse en el prompt del nodo `reason`, dandole al LLM informacion estructural:
- "Room-3 tiene la betweenness mas alta (0.45), es el cuello de botella de circulacion"
- "El cluster de MEP tiene baja closeness centrality, esta aislado del flujo principal"

### 4.4 Como exportacion complementaria al output final

El output actual del agente es un JSON de layout + score + explicacion. Agregar el grafo topologico (CSV de nodos/aristas + visualizacion) enriqueceria el deliverable final.

## 5. Conclusiones

1. **Compatibilidad de datos:** Ambos sistemas consumen el mismo JSON. La integracion es directa a nivel de datos, no se requiere conversion.

2. **Complementariedad analitica:** El agente analiza el espacio desde la perspectiva de un *usuario* (puede pasar, alcanzar, ver). El grafo analiza desde la perspectiva de la *red* (que es central, que conecta, que esta aislado). Son dimensiones ortogonales del mismo problema.

3. **El grafo resuelve un gap del agente:** El agente actual no tiene una vision global de la topologia del edificio. El nodo `reason` toma decisiones basandose en resultados parciales (collision de un room, path entre dos rooms). El grafo le daria una "vista de pajaro" de toda la estructura relacional.

4. **El agente resuelve un gap del grafo:** El graph generator es estatico y descriptivo. No propone cambios ni evalua accesibilidad. El agente aporta la capacidad de actuar sobre la informacion.

5. **Riesgo de ruido:** La relacion `spatial_near` con threshold de 15 unidades genera muchas aristas (especialmente en layouts densos, donde residential_02 llega a 0.961 de densidad). Para la integracion, seria mas util filtrar o ponderar estas relaciones, o usar solo las relaciones explicitas (references, contained_in, door_connection) que son mas semanticamente significativas.

6. **Integracion mas viable a corto plazo:** Usar las metricas de centralidad como contexto adicional para el Profile/Space Type agent o directamente en el prompt del reason node. No requiere cambios arquitectonicos grandes, solo ejecutar el graph generator como funcion auxiliar y inyectar los resultados en el estado del agente.

## 6. Retroalimentacion del profesor (Studio 2026-05-18, min 2:13:00 - 2:33:00)

### 6.1 La pregunta central del profesor

> "How do you search for a better position? [...] What is the mechanism? How does the LLM control the coordinates?"

El profesor identifico que el **gap principal** del agente es la falta de una **estrategia de busqueda** para el posicionamiento de objetos. Posicionar aleatoriamente y luego evaluar con scoring es "brute force" y "a bit dumb". Otro grupo tuvo exactamente este problema: entraba en un loop infinito porque el agente no sabia realmente que estaba haciendo.

### 6.2 La propuesta: el grafo como lenguaje del LLM

El profesor propuso usar un **grafo** como la representacion intermedia entre lo que el usuario pide y la realidad espacial. La idea central:

**Dos grafos que se comparan:**

1. **Grafo objetivo (target/optimal graph):** Codifica la estrategia del usuario. Cuando el usuario dice "quiero la CNC cerca de las maquinas sucias y junto a la fachada por ventilacion", esto se traduce a un grafo con:
   - Orden espacial de objetos (e.g., clockwise encoding desde un vertice)
   - Aristas de proximidad (CNC → dirty machines = "close")
   - Aristas de relacion funcional (CNC → facade = "adjacent")
   - Atributos de nodo (reachable: true/false, orientation, etc.)

2. **Grafo real (current/reality graph):** Codifica el estado actual del layout. Se actualiza cada vez que el agente mueve un objeto. Tiene las mismas aristas y atributos pero con valores reales.

**El mecanismo de busqueda:** En lugar de mover objetos al azar, el agente compara el grafo real contra el grafo objetivo y mueve objetos para **acercar el grafo real al objetivo**. El grafo es "the carrot in front of the donkey" — le da direccion al movimiento.

> "The graph becomes the language of the LLM. Every time the user says something, you adapt that graph, and you can always measure against reality and see how far off you are."

### 6.3 Los tools alimentan las aristas del grafo

El profesor enfatizo que **los tools existentes crean las aristas** que dan significado al grafo:

| Tool | Lo que aporta al grafo |
|------|----------------------|
| Collision | Aristas de conflicto espacial entre objetos |
| Visibility | Aristas de linea de vision entre pares de objetos |
| Path Analysis | Aristas de shortest path (distancia, accesibilidad) |
| Reachability | Atributos de nodo (reachable: true/false) |
| Orientation | Atributos de nodo (facing direction correcta o no) |

> "Your other tools are creating the edges that give meaning to this graph."

No se trata de reemplazar los tools, sino de que sus resultados se codifiquen como aristas y atributos en el grafo.

### 6.4 Por que NO reinforcement learning

Ambos profesores coincidieron:
- RL es "very technical" y "overcomplicated for what you need"
- Si se quiere mantener en la esfera del LLM, **la comparacion de grafos** es la via correcta
- El grafo ademas ayuda a escapar **local maxima** (un layout que puntua mejor que todo lo adyacente pero no es optimo globalmente), porque el mismo grafo puede materializarse en el espacio de multiples formas

### 6.5 Otras recomendaciones

- **NetworkX sobre TopologicPy:** "The simplest would be NetworkX." TopologicPy es bueno para inferir aristas faltantes, pero aqui todas las aristas son conocidas. Usar NetworkX.
- **Foco en industrial:** "Pick one to go deep into. The industrial one is actually the most interesting for you guys." Mantener residential como demo de que escala, pero profundizar en industrial.
- **Eficiencia de tokens:** "Make sure you're not sending information that's not relevant for that specific call." Las decisiones del LLM son simples; encadenar decisiones pequenas para obtener el resultado grande. No sobrecargar el system prompt.
- **Visualizar el proceso iterativo:** No solo mostrar before/after. Mostrar al usuario como el agente va probando posiciones y convergiendo. "Otherwise, the user goes, sends their prompt, and then just sits there for 5 min with no feedback."

### 6.6 Resumen ejecutivo del feedback

La pregunta focal para el equipo es:

> **"How do you come up with a strategy, and how do you search for implementing that strategy?"**

La respuesta propuesta: **El grafo es la estrategia.** Codifica lo que el usuario quiere como un grafo topologico (target), codifica la realidad como otro grafo (current), y la busqueda consiste en minimizar la diferencia entre ambos. Los tools existentes alimentan las aristas del grafo real. El LLM razona sobre grafos, no sobre coordenadas crudas.
