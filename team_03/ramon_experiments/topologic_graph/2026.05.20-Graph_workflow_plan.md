# Graph Dinámica de Relaciones Espaciales para Agente de Accesibilidad

## 1. Contexto del proyecto

Este proyecto consiste en un **agente de inteligencia artificial** cuyo objetivo es **analizar y mejorar la accesibilidad** en distintos tipos de plantas arquitectónicas (residenciales, comerciales, industriales).

El agente opera mediante un **workflow híbrido** que combina:

- **Grasshopper**: herramientas paramétricas para procesamiento geométrico y análisis espacial.
- **Python**: herramientas de cálculo, lógica de decisión y comunicación con el LLM.
- **Swiftlet**: puente de comunicación entre Grasshopper y el backend en Python.
- **LLM**: motor de razonamiento que ejecuta llamadas dentro de una graph de herramientas ya existente.

El estado actual del workflow ya cuenta con una **graph de orquestación de herramientas** (nodos = herramientas, edges = flujo de ejecución), que ejecuta loops adaptativos según la petición del usuario y el conocimiento acumulado durante la corrida.

## 2. Problema a resolver

La graph actual orquesta **herramientas**, pero no representa la **realidad espacial del proyecto**. No existe una estructura de datos que registre cómo se relacionan entre sí los **elementos arquitectónicos** (muros, puertas, ventanas, mobiliario, maquinaria, etc.) ni cómo evolucionan esas relaciones a lo largo del proceso de diseño.

Se necesita una **segunda graph**, distinta y paralela, que represente el **modelo espacial vivo** del proyecto.

## 3. Concepto: Graph dinámica de relaciones espaciales

### 3.1 Definición

Una graph donde:

- **Nodos**: cada elemento del archivo JSON del proyecto, identificado por su índice.
- **Edges**: relaciones espaciales, funcionales o normativas entre elementos (proximidad, orientación, dependencia, distancia mínima, alineación, frente a, etc.).

### 3.2 Comportamiento esperado

- La graph se **construye y actualiza automáticamente** a partir del archivo JSON.
- Cada vez que las herramientas del agente modifican propiedades del JSON, la graph **registra el cambio** y **reconfigura los edges** afectados.
- El agente usa la graph como **memoria contextual**: sabe qué relaciones existen, qué decisiones tomó previamente y cómo encajan los elementos entre sí.
- A medida que el JSON evoluciona, las decisiones del agente **mejoran iterativamente** porque cuenta con un mapa actualizado del estado del diseño.

### 3.3 Inferencia de relaciones nuevas

El agente debe poder **deducir relaciones** aunque las propiedades o nombres en el JSON no coincidan exactamente con las definiciones previas en el código. Ejemplos:

- Una **televisión** se relaciona con un **mueble de sala** mediante un edge tipo `frente_a`.
- Un **rack industrial** mantiene una distancia mínima con otros racks (edge `distancia_minima`).
- Una **máquina CNC** requiere ciertas herramientas y holguras alrededor (edges `requiere`, `holgura`).

Estas inferencias se incorporan a la graph como nuevos edges, ampliando el conocimiento del agente sobre el dominio.

## 4. Implementación propuesta (a validar)

### 4.1 Flujo de datos

1. El archivo **JSON** es la fuente de verdad del estado del proyecto.
2. Cada herramienta (Grasshopper o Python) lee y modifica el JSON.
3. Un **observador** sincroniza los cambios del JSON con la graph de relaciones.
4. El agente consulta la graph antes de tomar decisiones y la actualiza tras cada acción.
5. El ciclo se repite hasta cumplir el objetivo del usuario.

### 4.2 Coexistencia con la graph actual

- **Graph A (existente)**: orquesta herramientas, controla el flujo de ejecución del agente.
- **Graph B (nueva)**: representa el modelo espacial, sirve como memoria contextual.

Ambas graphs operan en paralelo y se comunican: la Graph A consulta la Graph B para tomar decisiones informadas.

## 6. Duda principal para discutir

¿Cómo integrar esta **Graph B (relaciones espaciales)** al workflow actual sin romper la **Graph A (orquestación de herramientas)**?

Puntos específicos a resolver:

1. ¿Dónde vive la Graph B? ¿En Python, en un archivo aparte, en memoria, en una base de datos ligera tipo SQLite o NetworkX serializado?
2. ¿Cómo se sincroniza con el JSON sin generar conflictos durante los loops del agente?
3. ¿Qué librería conviene para construirla y consultarla? (NetworkX, topologicpy, rustworkx, etc.)
4. ¿Cómo expone la Graph B su estado al LLM para que lo use como contexto? (serialización compacta, embeddings, descripciones en lenguaje natural).
5. ¿Cómo se conecta con topologicpy más adelante para la versión geométrica en Rhino?
